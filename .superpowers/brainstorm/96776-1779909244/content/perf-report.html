Paso 2: Análisis de Performance

Build de producción analizado. Estos son los problemas reales, ordenados por impacto.

🔴 Alto impacto

CSS de Swiper cargando en TODAS las páginas — y Swiper no existe más

app/layout.tsx importa 3 archivos CSS de Swiper globalmente. Pero SliderServices fue refactorizado a un grid CSS — Swiper ya no se usa en ningún lugar del proyecto. Son imports muertos que afectan el tiempo de carga de las 4 páginas.

Paquete motion instalado pero nunca usado

package.json tiene dos librerías de animación: framer-motion@11 (usada en todo el código) y motion@12 (cero imports en el proyecto). motion es el rebrand de framer-motion — tener ambos es peso duplicado.

🟡 Impacto medio

app/page.tsx tiene "use client" innecesario

El page component de Home no usa hooks ni event handlers propios. Todos sus hijos ya declaran su propio "use client". Sacarlo permite que Next.js trate Home como Server Component, reduciendo el bundle del route.

tsparticles no lazy-loaded — bloquea el render inicial

CoverParticles ya tiene un buen patrón de init lazy, pero el bundle de tsparticles se incluye en el chunk de la página. Con next/dynamic + ssr: false se puede diferir completamente del First Load JS.

Bug confirmado: dock tapa contenido — md:pb-10 insuficiente

Fix: container-page.tsx cambiar md:pb-10 → md:pb-36

🟢 Limpieza rápida

Remover @types/i18next de devDependencies

i18next v23+ incluye sus propios tipos. Este paquete es obsoleto y puede causar conflictos de tipos.
